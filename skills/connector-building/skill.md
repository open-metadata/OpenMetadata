# OpenMetadata Connector Building Skill

## When to Use

Activate when a user asks to build, create, add, or scaffold a new connector, source, or integration for OpenMetadata.

## Core Insight

**One JSON Schema definition cascades through 6 layers**: Python Pydantic models, Java models, UI forms (RJSF auto-render), API validation, test fixtures, and documentation. Define the schema once — everything else is generated or guided.

## Workflow: 7 Phases

### Phase 0: ENVIRONMENT — Set Up Python Dev Environment

Before any `make` or `python` commands, set up the environment from the repo root:

```bash
python3.11 -m venv env
source env/bin/activate
make install_dev generate
```

Always activate before running commands: `source env/bin/activate`

### Phase 1: SCAFFOLD — Generate Boilerplate

Run the interactive scaffold CLI to collect all inputs and generate files:

```bash
source env/bin/activate
metadata scaffold-connector
```

This walks through an interactive form collecting:
- **Connector name** (snake_case)
- **Service type**: database, dashboard, pipeline, messaging, mlmodel, storage, search, api
- **Connection type** (database only): sqlalchemy, rest_api, sdk_client
- **Auth types**: basic, iam, azure, jwt, token, oauth
- **Capabilities**: metadata, lineage, usage, profiler, stored_procedures, data_diff
- **Documentation URL** — API/SDK docs for the source
- **SDK package** — Python package name if applicable
- **API endpoints** — Key endpoints to call
- **Implementation notes** — Auth quirks, pagination, rate limits, etc.
- **Docker image** — If available, generates real testcontainers-based integration tests
- **Container port** — Port to expose from the Docker container

Or non-interactively:

```bash
metadata scaffold-connector \
  --name my_db \
  --service-type database \
  --connection-type sqlalchemy \
  --scheme "mydb+pymydb" \
  --auth-types basic \
  --capabilities metadata lineage usage profiler \
  --docs-url "https://docs.example.com/api" \
  --sdk-package "mydb-sdk" \
  --docker-image "mydb/mydb:latest" \
  --docker-port 5432
```

**Output**: All boilerplate files + a `CONNECTOR_CONTEXT.md` in the connector directory that any AI agent can read to implement the connector in one shot. When a Docker image is provided, integration tests use `testcontainers` to spin up a real instance.

### Phase 2: CLASSIFY — Understand the Source

The scaffold already classifies along 3 dimensions. Verify the choices:

**Dimension 1 — Service Type** (determines directory + base class):

| Service Type | Base Class | Reference Connector |
|---|---|---|
| `database` | `CommonDbSourceService` | `mysql/` |
| `dashboard` | `DashboardServiceSource` | `metabase/` |
| `pipeline` | `PipelineServiceSource` | `airflow/` |
| `messaging` | `MessagingServiceSource` | `kafka/` |
| `mlmodel` | `MlModelServiceSource` | `mlflow/` |
| `storage` | `StorageServiceSource` | `s3/` |
| `search` | `SearchServiceSource` | `elasticsearch/` |
| `api` | `ApiServiceSource` | `rest/` |

**Dimension 2 — Connection Type** (database only):
- `sqlalchemy` → `BaseConnection[Config, Engine]` + SQLAlchemy dialect
- `rest_api` → `get_connection()` + custom REST client
- `sdk_client` → `get_connection()` + vendor SDK wrapper

**Dimension 3 — Capabilities** (determines which extra files):

| Capability | Requires |
|---|---|
| `metadata` | Always present |
| `lineage` | Query log access (database) or dashboard→table links |
| `usage` | Query log access |
| `profiler` | SQLAlchemy connection |
| `stored_procedures` | Database-specific |
| `data_diff` | SQLAlchemy connection |

### Phase 3: RESEARCH — API/SDK Discovery

Read the `CONNECTOR_CONTEXT.md` generated by the scaffold. Then research:

**For Database (SQLAlchemy)**:
- SQLAlchemy dialect package and PyPI name
- Connection URL scheme (e.g., `postgresql+psycopg2`)
- Auth methods → map to existing `$ref` schemas (basicAuth, iamAuth, azureConfig)
- Query log source (system tables, slow logs, audit logs)
- Schema hierarchy (catalog → schema → table or just schema → table)
- System schemas to exclude by default

**For Database (REST/SDK)**:
- API documentation URL
- Auth type (token, OAuth, basic)
- Endpoints for listing databases, schemas, tables, columns
- Pagination pattern (offset, cursor, page token)
- Rate limits
- SDK package name

**For Dashboard**:
- API documentation URL
- Auth flow (OAuth, API key, basic)
- Endpoints for dashboards, charts, data models
- Project/workspace hierarchy
- Lineage: how charts reference data sources

**For Pipeline**:
- API documentation URL
- Endpoints for pipelines, tasks, runs, status
- Execution history access

**For Messaging**:
- Broker protocol (Kafka-like, AMQP, custom)
- Admin API availability
- Schema registry support
- Topic metadata access

### Phase 4: IMPLEMENT — Fill in the TODO Items

The scaffold generates files with `# TODO` markers. Implement them:

**`connection.py`** — The connection handler:
- Database/SQLAlchemy: Customize `_get_client()` if needed (custom URL building, auth token injection)
- Non-database: Implement `client.py` with REST/SDK calls, then wire `get_connection()` and `test_connection()`

**`metadata.py`** — The source class:
- Database: Usually works out of the box with `CommonDbSourceService`. Override methods only for custom behavior (stored procedures, custom type mapping)
- Non-database: Implement the abstract methods from the base class. Study the reference connector
- Non-database optional overrides: Lineage, usage, datamodel, ownership, and project/folder are implemented by overriding no-op methods in the base class — no extra files needed

**`client.py`** (non-database only):
- Initialize HTTP session or SDK client
- Implement `test_access()` for connection testing
- Add methods for each API operation needed

**`queries.py`** (database only):
- Add SQL queries for metadata extraction if the source needs custom queries
- Add query log SQL for usage/lineage extraction

### Phase 5: REGISTER — Integration Points

Modify these existing files to register the new connector:

| Step | File | Change |
|------|------|--------|
| 1 | `openmetadata-spec/.../entity/services/{serviceType}Service.json` | Add to type enum + connection oneOf |
| 2 | `openmetadata-ui/.../utils/{ServiceType}ServiceUtils.tsx` | Import schema + add switch case |
| 3 | `openmetadata-ui/.../locale/languages/` | Add i18n display name keys |

### Phase 6: GENERATE — Run Code Generation

```bash
source env/bin/activate                      # Activate Python environment
make generate                                # Python Pydantic models from JSON Schema
mvn clean install -pl openmetadata-spec      # Java models from JSON Schema
cd openmetadata-ui/src/main/resources/ui && yarn parse-schema  # Resolved JSON for UI forms
make py_format                               # Format Python code to mergeable state
mvn spotless:apply                           # Format Java code
```

### Phase 7: VALIDATE — End-to-End Checklist

```
[ ] JSON Schema: validates, $ref resolves, supports* flags correct
[ ] Code gen: make generate + mvn install + yarn parse-schema succeed
[ ] Connection: creates client, test_connection passes all steps
[ ] Source: create() validates config type, ServiceSpec is discoverable
[ ] Tests: unit + connection integration + metadata integration pass
[ ] Build: mvn spotless:apply, make py_format, make lint all pass
```

## File Patterns by Service Type

### Database (SQLAlchemy) — All Files

```
ingestion/src/metadata/ingestion/source/database/{name}/
├── __init__.py          # Empty
├── connection.py        # BaseConnection[Config, Engine] subclass
├── metadata.py          # CommonDbSourceService subclass
├── service_spec.py      # DefaultDatabaseSpec(metadata/lineage/usage/connection)
├── queries.py           # SQL query templates
├── lineage.py           # LineageSource mixin (if lineage capability)
├── usage.py             # UsageSource mixin (if usage capability)
├── query_parser.py      # QueryParserSource base (if lineage or usage)
└── CONNECTOR_CONTEXT.md # AI implementation brief
```

### Non-Database — All Files

```
ingestion/src/metadata/ingestion/source/{service_type}/{name}/
├── __init__.py          # Empty
├── client.py            # REST/SDK client wrapper
├── connection.py        # get_connection() + test_connection() functions
├── metadata.py          # {ServiceType}ServiceSource subclass
├── service_spec.py      # BaseSpec(metadata_source_class=...)
└── CONNECTOR_CONTEXT.md # AI implementation brief
```

### ServiceSpec Patterns

**Database**:
```python
from metadata.utils.service_spec.default import DefaultDatabaseSpec
ServiceSpec = DefaultDatabaseSpec(
    metadata_source_class=MyDbSource,
    lineage_source_class=MyDbLineageSource,     # if lineage
    usage_source_class=MyDbUsageSource,         # if usage
    connection_class=MyDbConnection,
)
```

**All others**:
```python
from metadata.utils.service_spec import BaseSpec
ServiceSpec = BaseSpec(metadata_source_class=MySource)
```
