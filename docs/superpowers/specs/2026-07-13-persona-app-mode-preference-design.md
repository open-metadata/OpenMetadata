# Persona App Mode Preference — Design

**Date:** 2026-07-13
**Branch:** `lyon`
**Related:** [openmetadata-collate PR #4903](https://github.com/open-metadata/openmetadata-collate/pull/4903) — introduces the `useAppMode` / `useAppRoutesRegistry` / localStorage-backed app-mode primitives that this design builds on.

## Problem

`useAppMode` (added in PR #4903) resolves the active UI mode from a Zustand store persisted to `localStorage` (`om.appMode`), defaulting to `"default"` (Classic). The Collate plugin registers `"ai"` at runtime via `useAppRoutesRegistry.registerRoutes('ai', …)` when `isDesktopApp || isAskCollateInstalled`.

Today there is no admin-authored default: every user sees Classic on first login and has to opt into AI mode themselves. We want an admin to be able to say, per Persona, "users with this default persona land in AI mode." When a persona doesn't specify a mode, the existing localStorage/default fallback must still work.

## Goals

1. Add an optional `appMode` field to persona-level UI customization so an admin can force Classic or AI per persona.
2. Provide an admin editor page — analogous to `SettingsNavigationPage` — under `CustomizablePage`.
3. On login (and when the user's default persona changes), if the persona specifies an `appMode`, write it into `useAppModeStore` so the app renders in that mode this session and in future sessions.
4. Keep OM core mode-agnostic: no hardcoded `"ai"` in core. Gate visibility of the whole feature on the app-routes registry containing at least one non-default mode.

## Non-goals

- User-facing header toggle to change app mode. Out of scope for this design.
- Per-user override on top of the persona default. Users already have `user.personaPreferences[]` in schema, but wiring an override here is a follow-up.
- Any BE business logic beyond the schema field. The BE just stores/serves the string; all resolution happens in the FE.

## Design

### 1. Schema — add `appMode` to `PersonaPreferences`

File: `openmetadata-spec/src/main/resources/json/schema/type/personaPreferences.json`

```json
{
  "properties": {
    "personaId": { … },
    "personaName": { … },
    "landingPageSettings": { … },
    "appMode": {
      "description": "App mode this persona forces on login (e.g. 'default', 'ai'). When unset, the user's own localStorage value (or the built-in default) takes over.",
      "type": "string"
    }
  }
}
```

Free-form `string`, not an `enum`. OM core stays mode-agnostic — the picker in the admin editor is populated from `useAppRoutesRegistry`, so any future mode a plugin registers (`"preview"`, etc.) is available without a schema change.

**Downstream code generation:**

- Java model regenerated by `mvn install` on `openmetadata-spec`.
- FE type `openmetadata-ui/…/generated/type/personaPreferences.ts` regenerated as part of `make generate`.
- No hand-written Java repository / resource changes: the field is passed through as part of `UICustomization.personaPreferences[]`.

### 2. Admin editor — new `SettingsAppModePage`

New file: `openmetadata-ui/src/main/resources/ui/src/pages/SettingsAppModePage/SettingsAppModePage.tsx`

Structure mirrors `SettingsNavigationPage.tsx`:

- Props: `{ onSave: (appMode: string) => Promise<void> }`.
- Reads current persona from `useCustomizeStore` and finds the matching entry in `document.data.personaPreferences[]` (by `personaId`).
- Local state: `selectedAppMode: string` seeded from that entry's `appMode` (or `DEFAULT_APP_MODE`).
- Available options built from `useAppRoutesRegistry`:
  - Classic (`DEFAULT_APP_MODE`, i.e. `"default"`) — always shown.
  - One entry per registered non-default mode. Labels come from an i18n key: `t('label.app-mode-<modeKey>')` with a fallback to the raw key.
- UI: `CustomizablePageHeader` + a `Card` containing an Ant Design `Radio.Group` (matches the visual weight of the navigation editor and stays inside the tokens documented in `openmetadata-ui/src/main/resources/ui/docs/colors.md`).
- `disableSave` compares `selectedAppMode` to the persisted value.
- `handleSave` calls `onSave(selectedAppMode)`.
- `handleReset` sets `selectedAppMode` back to `DEFAULT_APP_MODE`.

Guard: if `useAppRoutesRegistry` has no non-default modes registered, render an `ErrorPlaceHolder` explaining that no AI mode is available in this deployment. This makes the page safe even if a user URL-navigates in a deployment where the Collate plugin is absent.

### 3. Wire into `CustomizablePage`

File: `openmetadata-ui/src/main/resources/ui/src/pages/CustomizablePage/CustomizablePage.tsx`

Add a new switch case parallel to `'navigation'`:

```tsx
case 'app-mode':
  return <SettingsAppModePage onSave={handleAppModeSave} />;
```

Add a `handleAppModeSave` sibling to `handleNavigationSave` that:

1. Reads the current persona doc.
2. Updates or inserts the entry in `document.data.personaPreferences[]` matching the persona id, setting `appMode` on it while preserving `landingPageSettings`.
3. Persists via the same doc-store call the landing-page editor already uses.

The route entry (URL `/customize-persona/:fqn/app-mode`) is added wherever the existing `navigation` link is surfaced — the persona customize entry list. Same discoverability as "Customize Navigation."

### 4. Runtime resolution — force-write on persona load

New hook: `openmetadata-ui/src/main/resources/ui/src/hooks/useSyncAppModeFromPersona.ts`

```ts
export const useSyncAppModeFromPersona = (): void => {
  const currentUser = useApplicationStore(s => s.currentUser);
  const hasNonDefaultMode = useAppRoutesRegistry(
    s => Object.keys(s.routes).length > 0
  );

  useEffect(() => {
    if (!hasNonDefaultMode) return;

    const defaultPersonaId = currentUser?.defaultPersona?.id;
    if (!defaultPersonaId) return;

    const personaAppMode = resolvePersonaAppMode(defaultPersonaId);
    if (personaAppMode) {
      writeAppMode(personaAppMode);
    }
  }, [currentUser?.defaultPersona?.id, hasNonDefaultMode]);
};
```

`resolvePersonaAppMode` fetches the persona's `UICustomization` document (same call the landing-page code path already makes) and returns `personaPreferences.find(p => p.personaId === id)?.appMode`.

Mount the hook once inside `AppRouter` — the same component that already calls `useAppMode()`.

**Semantics:**

- Persona has `appMode` → we force-write it via `writeAppMode(…)`, which updates the Zustand store, its localStorage-persisted copy, and (via the existing `storage` listener) any other open tabs.
- Persona has no `appMode` → we do nothing. `useAppMode()` continues to return the persisted localStorage value, which itself defaults to `DEFAULT_APP_MODE`. This gives us the fallback chain the requirements describe: persona → localStorage → default.

**Why force-write rather than a runtime "read persona at call time" override:** every existing consumer already reads from `useAppMode`/`useIsAiMode`. Force-writing avoids threading persona lookup into every consumer, keeps the single source of truth in `useAppModeStore`, and reuses the existing cross-tab sync. The trade-off — an admin later unsetting a persona's `appMode` won't retroactively "un-force" a user's localStorage — is acceptable: the user will just keep the last mode they were forced into until they explicitly change it, which matches how any preference persistence normally works.

### 5. Visibility gate

The `hasNonDefaultMode` check above is the sole gate. It is applied:

- In `useSyncAppModeFromPersona` — no work if no plugin registered a mode.
- In the persona customize entry list — the "App Mode" link is hidden if `hasNonDefaultMode` is false.
- In `SettingsAppModePage` itself — placeholder if the URL is hit directly.

Core code contains no reference to `"ai"`, `askCollate`, or `isDesktopApp`. The Collate plugin's own `shouldEnableAiRoutes` decision remains the ground truth: if it doesn't register the mode, OM core doesn't offer the setting.

## Data flow

```
Admin edits persona in SettingsAppModePage
  → handleAppModeSave writes personaPreferences[].appMode into UICustomization doc
  → doc-store save

User logs in / defaultPersona changes
  → useSyncAppModeFromPersona effect fires in AppRouter
  → resolvePersonaAppMode fetches persona UICustomization
  → if appMode present, writeAppMode(personaAppMode)
    → useAppModeStore updated → localStorage 'om.appMode' updated → other tabs re-hydrate
  → AppRouter re-renders with the new mode's ModeRoutes
```

## Error handling

- Persona fetch fails: log via `showErrorToast` (same pattern as landing-page load), skip force-write, fall back to localStorage. Do not block login.
- `resolvePersonaAppMode` returns an `appMode` that isn't in the registry (plugin uninstalled since it was configured): `writeAppMode` still writes it, but `AppRouter` falls back to the default mode's routes (it already handles `state.routes[appMode]` being undefined). No crash; user sees Classic.
- Save from `SettingsAppModePage` fails: `showErrorToast`, keep local state so user can retry.

## Testing

**Unit:**

- `useSyncAppModeFromPersona.test.ts`:
  - No non-default mode registered → no-op even with a persona that has `appMode`.
  - Non-default mode registered, persona has `appMode: "ai"` → `useAppModeStore.currentMode` becomes `"ai"`.
  - Persona has no `appMode` → store unchanged from its persisted value.
  - Changing `defaultPersona` re-runs and updates the store.
- `SettingsAppModePage.test.tsx`:
  - Options list built from registry.
  - Save calls `onSave` with selected value.
  - Reset restores `DEFAULT_APP_MODE`.
  - Save disabled when selection matches persisted value.
- `CustomizablePage.test.tsx`: new `'app-mode'` case renders the new page and passes `handleAppModeSave`.

**Playwright:**

- Add scenario in the existing persona customization spec: admin sets a persona's app mode to AI, a user with that persona logs in, lands on the AI mode variant of a route the Collate plugin registers. (Only runs where the Collate plugin's routes are present; skipped otherwise.)

**Integration:** none required — no new BE endpoints or resources; the field flows through existing `UICustomization` CRUD.

## Rollout / migration

- Field is optional; existing persona documents remain valid.
- No DB migration needed. Existing JSON payloads without `appMode` deserialize with the field absent.
- Users in a deployment without the Collate plugin see no change — the gate hides all new UI.
